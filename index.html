Images

upload pipeline · shared spec across web · android · ios

Avatar

The member avatar shown on every Caller Card. Every channel center-crops and re-encodes locally so the server's normalizer never has to resize a well-behaved upload. The 1024 / 2 MiB caps stay as the hard backstop.

Setting Value
Output dimensions 512 × 512 (square)
Aspect center-crop to square before encode
Upload format image/jpeg, quality 0.85
Server allowlist PNG + JPEG recommended; GIF, BMP, TIFF, WBMP also accepted (legacy)
Server max dimension 1024 px (each side) — downscaled with bilinear interpolation, same format
Server max payload 2 MiB — typical 512² JPEG is 30–80 KB, well under

Per-channel implementation

Channel Crop + scale Source
Web crop modal · 512² canvas · toBlob('image/jpeg', 0.85) calmido-website/app/[locale]/(main)/settings/CropModal.js
Android centerCropSquare() · resize to 512² · JPEG @ 85 calmido-phone-android/.../ui/util/AvatarUtils.kt
iOS center-crop · render to 512² · jpegData(compressionQuality: 0.85) calmido-phone-ios/.../UI/Calmido/MemberEditScreen.swift

Server normalization

Live at domain-service/directory-service/.../converters/AvatarNormalizer.scala. Steps in order:

  1. Reject if payload > MaxBytes (2 MiB).
  2. Decode header; accept PNG / JPEG (recommended) plus legacy GIF / BMP / TIFF / WBMP.
  3. If either side > MaxDimension (1024), scale down preserving aspect ratio, re-encode in the same format.
  4. Return (format, bytes) — caller stamps mimeType from format.
Hero / background image upload spec — to be added when re-enabled.

Calmido Caller Card

shared 320×200 spec · web · android · ios

Canonical caller card used across all Calmido surfaces. Source of truth lives in calmido-website/components/card/CallerCard/; the Android and iOS implementations mirror this spec. Three equal rows (1fr each): top row carries the Calmido logo or a topLabel plus up to 3 flag icons; middle row carries the avatar; bottom row carries name + phone · location.

Card 320×200 · radius --r-lg (16px) · dotted background via .calmido_dotted_card · flag/icon glyphs are CSS masks driven by currentColor.

Name slot & per-variant labels

The name slot in the bottom row is always populated. Each variant ships a default English label (and localised counterparts) that is rendered as the fallback when no per-instance displayName is supplied. Only Shielded goes a step further: its label always wins, even when a displayName is set, so a leaked name from an upstream adapter can never surface on the card. The remaining variants — Member, Business, Neutral, Reported, Locked — show displayName when present and fall back to the label only when it is blank, so a caller can still see who the locked or reported person is.

Warning · shielded · locked

Fixed-avatar variants — middle-row glyph is a CSS mask, not an image. reported swaps to an orange→red gradient; alert is the shared red→deep-red template for locked / spoofing / scam / … — defaults render a locked member; override --cc-glyph (and --cc-glyph-size) to swap the glyph and drop in a caller-card__warning label next to the avatar to name the threat. The name slot then falls back to Anonymous/Unknown when no identity is set, like shielded and neutral. shielded uses near-black with bright-green ink; neutral uses grey.

Threat-band selection (web directory). On the number pages the flagged variant is chosen by the backend threat band — see domain-service/docs/rank-algorithm.md (a bounded 0–1 risk interval with a traffic-light band). Elevated (warning) → reported (orange, attention icon); High (dangerous) → the red alert template with the skull glyph, shipped on the web as caller-card--dangerous. Low / Unknown → not flagged. The band is decided from the interval's confidence bounds, so a lone unconfirmed complaint self-suppresses (stays Low) until corroborated.

Threat meter

community-threat bar (threat-bar) — one horizontal gauge shared by web, iOS & Android

Anatomy

A single left-origin gauge for the caller's community-threat level. Safe is on the left (green), dangerous on the right (red), so the raw 0–1 values map straight to a percentage. It is fed entirely by the backend threat interval on community.threat — see domain-service/docs/rank-algorithm.md — and carries no client-side heuristics, so every channel renders the same verdict. Three marks sit on the track: the gradient (the danger axis), the confidence band spanning [lower, upper] (tinted by the level), and a downward arrow at the mean.

Cutoffs

The band is gated on the lower bound of the interval (not the mean), so a wide, unconfirmed interval self-suppresses until the evidence tightens. Two thresholds — recalibrated on production data (domain-service/docs/rank-algorithm.md) — split the 0–1 axis into the three verdict zones. The arrow (mean) only positions the marker; it never decides the band.

0.45 0.72
Low Elevated High

lower ≥ 0.72 → High · lower ≥ 0.45 → Elevated · else Low · no interval → Unknown. Card mapping: Elevated → orange reported, High → red dangerous.

Implementation spec

Everything a native (iOS/Android) client needs to draw the same gauge from the DTO. All positions are a fraction of the track width; the axis origin is the left edge.

  • Input — community.threat = { mean, lower, upper, band }. mean/lower/upper ∈ [0, 1] on a danger axis (0 = safe, 1 = dangerous). band is the ordinal 0 Unknown · 1 Low · 2 Elevated · 3 High. When the whole object is absent, render nothing — never infer "safe" from missing data.
  • Geometry — arrow x = mean (clamp to [2%, 98%] so it stays on the track); band = left: lower, width: upper − lower (min width ~2px). All as a percentage of the track.
  • Track gradient (left→right) — --c-dark-green 0% → #9ACD32 30% → #F5A623 58% → --c-stop-red 100%.
  • Level colour (band tint + arrow) — Low --c-dark-green, Elevated #F5A623, High --c-stop-red, Unknown --c-mid-grey. Band is the level colour at ~38–45% opacity; arrow is the solid level colour.
  • Band fallback — prefer the server band. If it is absent, derive from the lower bound: lower ≥ 0.72 → High; lower ≥ 0.45 → Elevated; else Low; a null interval → Unknown. Gating on the lower bound means a lone unconfirmed complaint self-suppresses (stays Low) until corroborated.
  • Accessibility — expose the verdict as a label on the meter (e.g. "Risk level: Suspicious"); the band and arrow are decorative.

Slots & variants

Section card (.scard) — composition order, all slots optional:

  1. label — Calm_Label_S, bright-green, uppercase (or mono numeral, top-right, in --full)
  2. head — icon (40×40 tile, 56×56 in --full) + title + optional subtitle
  3. body — supports HTML (inline <strong>, <span>, <a>)
  4. cta — bright-green, underlined, arrow appended via ::after

Variants stack on the base .scard:

  • normal (default) — radius 16 · padding 24 · Calm_Title_M
  • --full — radius 24 · padding 32 28 · Calm_Title_L · hover lift · for feature pillars
  • --inverse — for use over dark surfaces (combinable with above)

Info card (.icard) — column primitive without chrome, all slots optional:

  1. icon — 28×28 currentColor mask (or any inline SVG/img); margin-bottom 4 separates from text
  2. label — Calm_Label_S, eyebrow above the title
  3. title — Calm_Title_M, primary ink (or white in --inverse)
  4. subtitle — 14/20 Open Sans 600, Dark Green (or Bright Green in --inverse); sits between title and body
  5. body — Calm_Body_L, supports HTML
When a card has a cta and href, render .scard as <a> (linked, hover state). Otherwise <div> (static). Use .icard inside a multi-column grid (e.g. content_section_grid + grid_column_4) when no chrome (background, border, padding) is needed.
